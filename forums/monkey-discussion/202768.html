ziggy(Posted 2012) [#23]
It makes sense. Monkey is much closer to Java or C# or even .net than any of the previous Blitz products. The language complexity and flexibility make Monkey a very different product compared to any traditional Basic dialect. And also, They're presenting something new, to a new market, so it makes sense to be coherent and have a new page. Not even BRL is mentioned in the Monkey page AFAIK.

In my personal experience with Monkey, when I was in the beta test dev team for the Monkey development, and I was designing Jungle Ide:

I first tried to make it as an extension of the currently existing BlitzMax BLIde parser. then:
* Datatype inference --> Need to rewrite sensible parts of the parser
* Case sensitive language --> Ok, need to rewrite other areas that involve mostly any identifier matching algo.
* Very different Imports structure --> Ok, another rewrite
* AST building differences --> Ok, this has to be changed too
* Compiler interoperation --> Very different, better change it instead of extend it...

At the end... It felt I was really creating a parser and IDE for a new language and not for the natural evolution of a pre-existing BlitzMax, and that was what decided me to make a separated IDE for Monkey, and I'm happy I did becouse I could take the oportunity to redo all the areas that I wish I had done different when writing BLIde.

Now I'm creating an interpreted language called Harpl http://code.google.com/p/harpl-project/ and i'm writing all the compiler, runtimes abd intermediate code in Monkey, and I have had no issues with it. It's ultra powerful. I just wish it had delegates and it did not use the < and > for generics. It's a very bad idea to use the same symbol for different tasks if you want a language to be easy to parse (that was avoided in D for exactly the same reason and I can only say it was a wise decision). Other than that, for me, it's perfect, and rock solid.
