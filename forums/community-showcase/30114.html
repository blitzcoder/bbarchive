Game-timming for 60hz : here is what I am using for my next game.
Option00=0, set your directX to 60hz. The game buffer will flip with the monitor ( perfect )
Option00=1, slows the game by the time elapsed ( not good ) between too frames.
Option00=2, alternative method, it will "count" your refresh rate every seconds and make a "smoth" variable. The game could "fake" a 50hz or 60hz method, so the game could run a little bit slower depending of your monitor refresh rate ( good ).

Put this at the begining of the main loop :


NIPS00#=0.0;nombre d'image par seconde
NIPS01#=0.0;millisecs() du jeu
NIPS02#=0.0;millisecs() du jeu Tampon
NIPS03#=0.0;difference entre les deux tampons
NIPS04#=0.0;calcul du nombre de boucle par seconde approximatif
NIPS05#=0.0;calcul du nombre de boucle par seconde approximatif tampon

SpriteMove#=0.0

Option00%=0;select your game timming mode

Select Option00
	;*******************************************************************
	Case 0;60hz
	;mise a jour
	NIPS00=1.0
	;*******************************************************************
	Case 1;normal
	;calcul du nombre d'image par seconde
	NIPS01=MilliSecs();mise a jour
	NIPS03=(NIPS01-NIPS02);difference entre millisecs() Tampon et millisec()
	;mise a jour du tampon
	NIPS02=NIPS01;mise a jour millisecs() Tampon
	;mise a jour
	NIPS00=(NIPS03/(1000.0/60.0))
	If NIPS00<0.0 NIPS00=0.0
	If NIPS00>1.0 NIPS00=1.0
	;*******************************************************************
	Case 2;60hz alternatif
	;mise a jour
	NIPS01=MilliSecs()
	NIPS04=(NIPS04+1)
	;mise a jour toutes les secondes
	If NIPS01>=NIPS02
		NIPS05=NIPS04
		NIPS04=(-1.0)
		NIPS02=(MilliSecs()+1000.0)
		;mise a jour
		NIPS03=(60.0/NIPS05) Mod 0.25
		NIPS00=((60.0/NIPS05)-NIPS03)
		;mise a jour
		If NIPS00<0.0 NIPS00=0.0
		If NIPS00>1.0 NIPS00=0.0
	EndIf
End Select

;moving sprite 1 pixel per loop
SpriteMove=(SpriteMove+(1.0*NIPS00))

Réno - I understand... I love good pixel art too, although these days my own ideal is to make things look like "proper" claymation as much as possible so I'll take any tricks I can get... plus, without vertex colours and scaling/rotation my gfxmem requirements would go through the roof! Using scaling, rotation and transparency to me is also much more like the 2d work I am used to as an animator; using aftereffects etc. I reckon it was the polygon meshes that killed 2d - plus the rather faddy idea that every game now needs a 3d camera that can view the action from any angle.

But far better to be a hardcore pixel art fan than yet another "only 3d is acceptable" person as far as I'm concerned. But I think ironically, the novelty of 3d has worn off and nowadays 2d is the novelty. I think we may see a minor resurgence of 2d as developers try to make their games look different from everyone else's (finally!)

As for the timing, I am running internal logic every 3ms, and rendering a frame whenever I get a vblank. Works pretty well, but stutters occasionally; I think because vwait is too resource hungry. I'll look into making that side of things a little more intelligent later and see if I can't get it totally silky smooth.

Erik - I like em too... but I don't think I will include an idle demo this time around. However, I am definitely considering recording all the players moves and if they play a record breaking round on any given level, saving the replay for posterity. If I do have an idle demo, it will play back the player's own best performances.
